Auth Screens: Login & First-Run Admin Setup

Two auth flows. Simple but sets the visual tone for the whole app.

Flow 1: Login

POST /api/auth/login → JWT → store in cookie → redirect to /projects. Standard.

Flow 2: First-Run Admin Setup

Backend seeds an admin user on first boot with a known default password (TestAdmin123! in dev). In production, the admin password should be set via env var. The "first-run" page is a forced password-change screen: detect default credentials → force new password → then login normally.

Question: should this be a dedicated /setup page, or just part of the login flow?

A

Dedicated /setup Page

Separate route. Detects first-run (no users or admin has default password). Full-page form: new admin password + confirm. On success → redirect to /login.

Clear separation. Can be skipped once setup is done. Matches most self-hosted tools (Gitea, GitLab).

B

Inline in Login Page

Single /login page. If first-run detected, show "Create Admin Account" form. Otherwise show normal login form.

Fewer routes. One component with conditional rendering. Less clear for users expecting a login page.

C

No First-Run Page — Env Var Only

Admin password set via env var at deploy time. No in-app setup flow. Login page only.

Simplest. But if env var isn't set, admin is locked out. Less user-friendly for self-hosted adopters.

Login Page Visual Style

Centered card on all sizes. Logo + title "HydraForge" at top. Username + password fields + "Sign in" button. Error below form (toast-adjacent, not floating toast). Dark/light toggle in corner.

Recommendation: A

Dedicated /setup page. Standard self-hosted pattern. Clean separation of concerns. After setup, route is inaccessible (redirect to /login). Simple middleware: check first-run state on app load.
