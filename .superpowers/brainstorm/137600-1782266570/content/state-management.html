State Management: How Does Board Data Flow?

Server is authoritative. SignalR pushes updates. How does the client hold and react to state?

The data flow problem

User moves card → POST to API → server broadcasts via SignalR → all clients update. But what happens between POST and SignalR arrival? Optimistic update? Wait for server? And where does the board state live — Pinia store, Nuxt useState, or just refs in components?

A

Pinia Store + Optimistic Updates

Single Pinia store holds board state (columns, cards). Mutations: optimistic local update → POST to API → on success, do nothing (SignalR will confirm). On error, rollback + toast.

SignalR events update the store directly. Components react via store getters.

Snappy UX. Risk: brief flicker if optimistic state differs from server state. Pinia = extra dep but well-established.

B

Server-Confirmed Only (No Optimistic)

POST to API → wait for 200 → update local state. SignalR events update state for other users' changes. No optimistic updates — UI shows loading indicator during mutation.

State lives in composables using Nuxt useState (no Pinia).

Simpler. No rollback logic. Feels slower on high-latency connections. Always consistent.

C

Hybrid: Optimistic for Moves, Confirmed for Edits

Card moves (drag-and-drop) = optimistic (instant feedback matters). Card edits (title, description) = server-confirmed (less urgency).

Pinia store. Moves: optimistic + rollback on 409. Edits: loading state → update on 200.

Best UX where it matters. More complex — two code paths for mutations.

SignalR integration

Regardless of option: @microsoft/signalr npm package. Connect on project page mount, disconnect on leave. BoardHub events → update store. PresenceHub → online users list.

Reconnection: SignalR auto-retry with backoff. Show "Reconnecting..." banner during disconnect.

Recommendation: A

Pinia + optimistic updates. Drag-and-drop without instant feedback feels broken. Rollback on error is a well-understood pattern. Pinia is 2KB, worth the dep for structured state. SignalR events just call store actions — clean separation.
